# Проект высоконагруженного сервиса Сбербанк Онлайн
## 1. Тема и целевая аудитория
### Описание и аналоги
#### Описание [[1]](https://www.sberbank.ru/ru/person/dist_services)
Сбербанк Онлайн — удаленный канал обслуживания Банка, автоматизированная защищенная система дистанционного обслуживания Клиента через Официальный сайт Банка в сети Интернет, а также Мобильное приложение Банка. Сбербанк Онлайн позволяет получать подробную информацию о банковских услугах: счетах, вкладах, банковских картах и кредитах, а также погашать кредиты, совершать платежи, переводы между вкладами и счетами банковских карт, переводить деньги клиентам Сбербанка и других банков. Помимо этого, в канал обсуживания интегрировано большое количество иных сервисов Сбербанка, включая сервисы для инвестирования, получения документов, контроля и оплаты услуг ЖКХ, использования развлекательных сервисов экосистемы Сбера и пр.
Сервис работает как в виде интернет-кабинета через браузер, так и в виде мобильного приложения для Android и iPhone.

#### Аналоги [[2]](https://www.rustore.ru/prostore/banki-obzor-prilozhenii-1)
У сервиса есть ряд аналогов – как зарубежных, так и отечественных. Перечислены сервисы, которые также интегрировали большое количество услуг в свои приложения, помимо тех, которые напрямую относятся к управлению финансами. Основные конкуренты в настоящий момент — отечественные (работа россиян с зарубежными банками осложнена): ТБанк, ВТБ Онлайн, Альфа Банк, Озон Банк, Почта Банк, Точка Банк и др.

### Объём хранилища
+ Среднее количество транзакций в минуту:
  `34 722 + 1 526 + 64 061 = 100 309`
+ Средний размер одного чека равен 60 кБ. Тогда за минуту генерируется:
  `36 248 * 60 кБ = 2 174 880 кБ = 2,074 ГБ` информации, которая заполняет память
+ Объём данных, генерируемых за день транзакциями (сделаем поправку 0,74 на ночное время в часовом поясе МСК) [[12]](https://ru.wikipedia.org/wiki/Население_России):
  `2,074 ГБ * 60 * 24 * 0,74 = 2 210,0544 ГБ`
+ Ежемесячный объём данных по транзакциям:
  `2 210,0544 ГБ * 30 = 66 301,632 ГБ = 64,75 ТБ`
+ Годовой объём данных по транзакциями:
  `64,75 ТБ * 12 = 777 ТБ`
+ Также данные пользователей хранятся в профилях весом 512 кБ:
  `700 кБ * 84 300 000 = 659,48 ТБ`
+ Всего:
  `777 ТБ + 659,49 ТБ = 1436,49 ТБ = 1,4 ПБ`

+ 1.5 - коэффициент увеличения пути (оптоволокно не идет по прямой) [[14]](https://books.google.ru/books?id=5LMp7yxfeDAC&source=ttb&redir_esc=y),
+ 200 - скорость распространения сигнала в оптоволокне (км/мс) [[15]](https://nag.ru/news/18705),
+ 1 мс - статистическая погрешность -> запас на задержки сетевого оборудования.

### Выбор схемы балансировки
Зачастую при реализации высоконагруженных систем используется комбинация нескольких схем балансировки: Anycast, DNS и, при наличии ресурсов, HTTP [[18]](https://habr.com/ru/companies/oleg-bunin/articles/979582/). Цель совместного применения различных типов в высоконагруженных банковских приложениях заключается в обеспечении непрерывной доступности приложений и минимизации времени ожидания ответов от них [[19]](https://www.edgenexus.io/ru/блог/как-финансовые-учреждения-могут-пред/). Ниже приведено описание схем балансировки, а также их сочетание, которое будет оптимальным для приложения Сбербанк Онлайн:
1. **DNS-балансировка**: функциональные домены data, products, payments  
**DNS-балансировка** – это объявление нескольких IP-адресов для одного доменного имени с целью равномерного распределения входящего на сервер трафика и обеспечения минимального времени задержки при выполнении запроса пользователем [[20]](https://www.geeksforgeeks.org/computer-networks/dns-load-balancing-round-robin-global-server-load-balancing/).

Наиболее распространёнными типами DNS-балансировки являются Round Robin (последовательное распределение DNS-запросов по пулам), Weighted Round Robin (усовершенствованная версия алгоритма Round Robin, в котором нагрузка распределяется равномерно с учетом веса серверов: серверы с большим весом обрабатывают больше запросов) и Failover Group (алгоритм, при котором запросы отправляются на сервер с наибольшим весом до тех пор, пока он доступен) [[21]](https://www.hostwinds.ru/blog/what-is-dns-load-balancing)[[22]](https://cloud.vk.com/docs/networks/dns/concepts/global-balancer). Одним из основных преимуществ является возможность кэширования данных сессии в течение установленного времени (Time to Live, TTL), что позволяет не выполнять громоздкие запросы данных каждый раз, когда они необходимы [[20]](https://www.geeksforgeeks.org/computer-networks/dns-load-balancing-round-robin-global-server-load-balancing/). Тем не менее, эта характеристика приводит и к ограничениям продуктивности метода – если в течение какой-либо из серверов упал, пользователь узнает об этом только по прошествии TTL, а до этого момента будет получать ошибку в работе системы; кроме того, большой объём кэша замедляет скорость работы приложения [[21]](https://www.hostwinds.ru/blog/what-is-dns-load-balancing). В связи с этим DNS-балансировку оптимально использовать для нагруженных доменов, работа с которыми требует запроса большого количества данных, но при этом не имеет критическую зависимость от скорости реакции на изменения в структуре доступных серверов и имеет возможность поддерживать длительные бесперебойные соединения: среди них в приложении "Сбербанк Онлайн" можно выделить data, products и payments.
 
Использовать предлагается метод балансировки GeoDNS, т.к. этот метод, в отличие от Round Robin и Weighted Round Robin, учитывает географическое положение клиента, который делает запрос, и предоставляет ответ с ближайшего доступного сервера, тем самым равномерно распределяя нагрузку между ними и снижая время ожидания для пользователей. DNS-серверы Сбера (GSLB) постоянно контролируют состояние балансировщиков в каждом ЦОД, и если ближайший к пользователю ЦОД недоступен или перегружен, то запрос автоматически перенаправляется на следующий оптимальный ЦОД. Учёт этих факторов особенно важен при работе на территории России, которая обладает огромной территорией и неравномерным расселением людей (почти 75% населения проживает в европейской части страны, которая составляет всего 25% от общей площади государства [[23]](https://ru.wikipedia.org/wiki/Европейская_часть_России)).

**Схема DNS-балансировки**
1. Клиент (мобильное приложение / web-версия) получает IP-адрес устройства пользователя.
2. Запрос поступает в DNS-серверы Сбера, которые определяют примерное местоположение устройства.
3. Клиенту возвращается IP-адрес балансировщика ближайшего ЦОД.
4. В целях обеспечения отказоустойчивости, DNS-записи имеют небольшое время жизни. Это позволит в случае аварии быстро перенаправить трафик, изменив DNS-ответы.

**Схема:** Клиент -> DNS-запрос -> DNS с GeoIP -> Ответ (IP ближайшего ЦОД) -> Клиент -> Запрос к этому IP.

2. **Anycast-балансировка**: функциональные домены auth
**Anycast-балансировка** – это метод балансировки, при котором один и тот же IP-адрес анонсируется из нескольких серверов, географически удалённых друг от друга, и система автоматически направляет запрос пользователя к серверу, к которому ведёт ближайший сетевой путь [[24]](https://kingservers.com/blog/bgp-anycast-dlia-ghlobalnoi-dostavki-trafika-kak-eto-rabotaiet-iznutri/). Эта балансировка происходит на сетевом уровне (OSI L3) и наиболее распространённым протоколом определения сетевого маршрута трафика является Border Gateway Protocol (BGP) [[18]](https://habr.com/ru/companies/oleg-bunin/articles/979582/). Ключевым преимуществом этого вида балансировки является возможность быстрого перенаправления трафика – при выключении одного из серверов из сети, IP-адрес перестаёт отражаться на нём, и система сразу меняет маршрут на наиболее оптимальный из доступных [[24]](https://kingservers.com/blog/bgp-anycast-dlia-ghlobalnoi-dostavki-trafika-kak-eto-rabotaiet-iznutri/). В связи с этим, подобный метод балансировки удобнее всего применять для работы с доменами, для которых критически важны бесперебойная работспособность и минимальное время ожидания ответа, при этом способность поддерживать длительные соединения не требуется. Среди таких функциональных доменов в модели приложения "Сбербанк Онлайн" можно выделить auth (авторизация).

**Схема Anycast-балансировки**
1. Один и тот же IP-адрес (например, для серверов авторизации) поступает из нескольких ЦОДов одновременно (Москва, Балаково) через протокол BGP.
2. Маршрутизаторы в интернете при помощи протокола BGP направляют запрос клиента к тому ЦОДу, который ближе с точки зрения маршрутизации.

**Схема:**  Пользователь -> Запрос к Anycast IP -> Интернет-маршрутизация (BGP) -> Ближайший доступный ЦОД -> Внутренний балансировщик -> Бэкенд

3. **HTTP-балансировка**: может применяться для функциональных доменов data и products
Данный вид балансировки используется для оптимального распределения пользователей по точкам входа в приложение. Существует два основных типа: перенаправление на свободные хосты и подмена ссылок в передаваемых ответах [[18]](https://habr.com/ru/companies/oleg-bunin/articles/979582/). HTTP-балансировка (L7) анализирует содержимое HTTP-запросов (заголовки, cookie, URI) и позволяет направлять трафик к конкретным сервисам в зависимости от типа операции. Это ресурсоёмкая технология, но она может быть особенно полезна для доменов data и products, где требуется гибкая маршрутизация (например, отдельные пулы серверов для просмотра истории и для скачивания выписок).

В настоящее время Сбербанк реализовал сетевой балансировщик нагрузки LBOSSE, который объединяет в себе различные методы балансировки на уровнях OSI L4–L7 [[25]](https://reestr.digital.gov.ru/reestr/2456658/). К примеру, при подключении по L3 c использованием BGP при выпадении всех серверов из группы балансировки прекращается анонсирование соответствующих виртуальных IP (Anycast), а также есть встроенные механизмы манипуляции с трафиком http/https, в т. ч. вставка тегов, анализ и замена header’ов и URI (HTTP) [[26]](https://www.sberbank.ru/ru/legal/services/lbosse).

### Механизм регулировки трафика между ЦОД
1. GSLB (Global Server Load Balancing) на основе DNS
Система GSLB постоянно контролирует состояние и нагрузку на всех балансировщиках в ЦОД.
Алгоритмы:
+ GeoProximity: по умолчанию направляем клиента в ближайший ЦОД;
+ Оценка уровня загрузки: если ближайший ЦОД занят, перенаправляем на следующий
2. Управление с помощью функциональных доменов
+ На уровне балансировки внутри ЦОДа, мы можем регулировать трафик запросов к конкретным микросервисам. Например, прием сложных операций по оформлению кредитов (домен products) можно перенаправлять из ЦОД "Южнопортовый" в более мощный ЦОД "Сколково".

## 4. Локальная балансировка нагрузки
При росте объёма трафика и увеличении нагрузки на серверы приложения в архитектуру системы добавляется балансировщик, который позволяет эффективно распределять входящую нагрузку между компонентами кластера. Такой процесс называется локальной балансировкой; он обеспечивает отказоустойчивость приложения, оптимальное использование ресурсов, плавное масштабирование инфраструктуры, а также может выполнять функцию защиты от DDoS-атак [[1]](https://selectel.ru/blog/load-balancer-review/).

### Основные положения балансировки
Зачастую локальная балансировка в высоконагруженных приложениях в т. ч., банковских) происходит на уровнях L4 и L7 OSI [[2]](https://timeweb.com/ru/community/articles/kak-ustroen-balansirovshchik-nagruzki). Такую же балансировку планируется применять и для приложения Сбербанк Онлайн:
1. L4-балансировка (Транспортный уровень): распределение TCP/UDP-трафика между серверами с наименьшим уровнем нагрузки на основаниии анализа сетевой информации о трафике (протоколы, порты) [[1]](https://selectel.ru/blog/load-balancer-review/), [[3]](https://www.computerra.ru/311930/peredacha-dannyh-setevye-tehnologii-ot-ethernet-do-ipv6/). Позволяет быстро распределять весь входящий трафик, так как не требуется доскональное изучение данных [[4]](https://www.geeksforgeeks.org/system-design/layer-4l4-layer-7l7-and-gslb-load-balancers/).
2. L7-балансировка (Прикладной уровень): распределение HTTP/HTTPS-трафика на основании содержимого запросов, HTTP-заголовков и cookie-файлов [[1, 3]]. Кроме того, L7-балансировщики могут выполнять терминирование SSL [[4]]. Балансировка данного уровня применяется тогда, когда необходимо настроить маршрутизацию по специализированным серверам [[4]].

В проекте предлагается использовать следующую связку:
- **L4:** HAProxy / Linux Virtual Server (LVS) в режиме TCP-балансировки.
- **L7:** NGINX Ingress Controller в кластере Kubernetes для терминирования SSL и направления запросов к подам сервисов.

Дополнительно можно отметить, что Сбер разработал и внедрил собственный балансировщик нагрузки **Platform V SynGX** [[5]](https://www.tadviser.ru/index.php/Проект:Сбербанк_%28СберТех:_Platform_V_SynGX%29), способный работать на уровнях L4 и L7, с полной интеграцией в Kubernetes и поддержкой аппаратного ускорения SSL. Это решение может использоваться как альтернатива NGINX Ingress Controller в промышленной среде банка.

#### 4.2. Схема балансировки и механизм резервирования
Каждый уровень балансировки резервируется по формуле **N+1**, где N — минимальное количество узлов, необходимое для обслуживания пиковой нагрузки. Все узлы работают в активном режиме, равномерно распределяя нагрузку. При отказе одного узла оставшиеся продолжают обслуживание, так как их совокупной производительности достаточно для полной нагрузки.

- **L4-балансировщики:** два или три узла HAProxy/LVS с общим виртуальным IP (keepalived). Активная‑активная конфигурация.
- **L7-балансировщики (Ingress NGINX):** развёртываются как Kubernetes Deployment с необходимым количеством реплик. Поды равномерно распределяются по разным рабочим (worker)-узлам, чтобы исключить единую точку отказа на уровне железа.

### 4.3. Расчёт количества L7-балансировщиков (Ingress NGINX)
Расчёт выполнен на основе официальных тестов производительности NGINX:

- **SSL TPS (транзакций в секунду):** ~56 700 при 16 vCPU, RSA 2048 [[6]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes).
- **HTTPS RPS:** ~340 000 при 16 vCPU, keep-alive соединения [[7]](https://blog.nginx.org/blog/testing-the-performance-of-nginx-and-nginx-plus-web-servers).
- **Пропускная способность:** 10 Гбит/с на один экземпляр.

**Исходная нагрузка на один ЦОД** (на примере ЦФО, ~50% общего трафика):

| Показатель | Значение |
|------------|----------|
| Пиковый общий RPS | 12 933 |
| Пиковый SSL TPS (новые соединения, домен auth) | 1 531 |
| Пиковый сетевой трафик | 0,61 Гбит/с |

**Минимально необходимое количество (N) по видам ограничений:**

| Ограничение | Производительность одного NGINX | Требуемая мощность | Расчётное кол‑во |
|-------------|----------------------------------|--------------------|------------------|
| SSL TPS | 56 700 TPS | 1 531 TPS | 1 |
| HTTP RPS | 340 000 RPS | 12 933 RPS | 1 |
| Пропускная способность | 10 Гбит/с | 0,61 Гбит/с | 1 |

Таким образом, согласно данным о производительности, достаточно одного балансировщика. Однако с учётом требований к отказоустойчивости и возможности проведения планового обслуживания принимаем **N=2** (два узла полностью перекрывают пиковую нагрузку, так как один экземпляр способен выдержать полный трафик ЦОД). Резервирование по формуле **N+1** даёт 3 реплики Ingress NGINX в каждом крупном ЦОД.

Все три реплики активны одновременно. При отказе одной оставшиеся две продолжают работу, при падении двух — одна оставшаяся всё ещё способна обработать всю нагрузку, что подтверждается расчётами запаса по SSL TPS и RPS.

**L4-балансировщики** также развёртываются в количестве 3 экземпляров (N = 2 + 1) с виртуальным IP, обеспечивая аналогичную отказоустойчивость на транспортном уровне.

### 4.4. Итоговая схема резервирования
- **Уровень L4:** 3 × HAProxy/LVS, актив‑актив, N+1.
- **Уровень L7:** 3 × NGINX Ingress Controller (или Platform V SynGX), актив‑актив, N+1.
- **Отказоустойчивость:** при выходе из строя любого узла оставшиеся автоматически перераспределяют нагрузку; при полном отказе двух L7-узлов оставшийся один справляется с пиковым трафиком.

### Детали по кэшам и временным файлам
#### UserCache
- **Назначение**: хранение часто запрашиваемых данных пользователя (профиль, список счетов, карт, настроек) для снижения нагрузки на основную БД.
- **Реализация**: Redis Cluster, шардирование по `user_id`.
#### SessionCache
- **Назначение**: хранение данных, специфичных для текущей сессии: черновики платежей, временные OTP-коды, состояние двухфакторной аутентификации.
- **Реализация**: отдельный Redis Cluster, шардирование по `session_id`. Жизненный цикл привязан к сессии.
#### TempFiles
- **Назначение**: временные файлы, создаваемые пользователем (загруженные изображения для платежей, предварительные квитанции, черновики документов).
- **Реализация**: метаданные хранятся в реляционной БД (или NoSQL), файлы — в объектном хранилище. Поле `expires_at` определяет автоматическое удаление.

### Требования к консистентности
- **Сильная (Strong)**: обязательна для всех сущностей, связанных с деньгами и счетами (`Users`, `Accounts`, `Cards`, `Transactions`, `Payments`, `Loans`, `Deposits`). Используются реляционные СУБД с поддержкой ACID.
- **Eventual**: допустима для:
  - истории и логов (`OperationHistory`, `Notifications`);
  - кэшей (`UserCache`, `SessionCache`) — потеря кэша не критична, т.к. данные восстанавливаются из основной БД;
  - временных файлов (`TempFiles`) — файл может появиться в хранилище чуть позже записи метаданных, что допустимо;
  - чатов поддержки (`SupportChats`, `ChatMessages`) — небольшие задержки репликации не влияют на пользовательский опыт;
  - выписок (`Statements`) — генерация документа может быть асинхронной.

### Распределение нагрузки по ключам
- **Основной ключ шардирования** — `user_id` (хэш) для большинства сущностей, обеспечивая локальность данных одного клиента.
- **Транзакции и платежи** дополнительное партицирование по дате для эффективной архивации и удаления старых данных.
- **Сессионный кэш** шардируется по `session_id`, что позволяет равномерно распределять нагрузку между узлами Redis.
- **Географическая репликация**: данные критических сущностей реплицируются между ЦОД с контролем согласованности. Кэши и временные файлы могут иметь локальную копию в каждом ЦОД для минимизации задержек.

Партицирование по дате + сортировка по (user_id, timestamp) позволяет эффективно выполнять агрегатные запросы (например, «история операций за месяц»).|

## 10. Схема проекта
<img width="2973" height="1724" alt="image" src="https://github.com/user-attachments/assets/3e1ebdd6-60eb-4dc0-9b2b-1302c39e35bc" />

**Пояснение к схеме:**  

- **GeoDNS:** глобально маршрутизирует запросы к сервисам данных, платежей и продуктов в ближайший доступный ЦОД с учётом географической близости и загрузки.
- **Anycast BGP:** обеспечивает вход в сервис авторизации с минимальной задержкой и мгновенное переключение при отказе ЦОД.
- **L4-балансировщик (HAProxy)**: принимает TCP-соединения из интернета и равномерно распределяет их на L7-балансировщики, чем обеспечивает защиту от сетевых атак.
- **L7 Ingress (NGINX):** терминирует SSL, анализирует HTTP-заголовки и маршрутизирует запросы к нужным микросервисам (auth, payments, data, products), выполняет rate limiting.
- **auth-service:** производит аутентификацию пользователей, создаёт и валидирует сессии. Также хранит токены в общем SessionCache Redis, откуда их проверяют все остальные сервисы.
- **payments-service:** выполняет переводы и платежи (СБП, карты, QR), управляет распределёнными транзакциями по алгоритму Saga, записывает данные в PostgreSQL и отправляет события в Kafka для уведомлений и обновления кэша.
- **data-service:** обслуживает запросы просмотра счетов, карт, профилей, истории операций; читает из PostgreSQL и ClickHouse, активно использует Redis (UserCache) для кэша и горячих данных.
- **products-service:** оформляет кредиты и вклады.
- **statement-generator:** асинхронно создаёт PDF‑выписки по заданиям из Kafka, сохраняет файлы в S3 и обновляет статус в PostgreSQL.
- **notification-service:** получает события о готовности документов или важных действиях, отправляет push‑уведомления пользователям, фиксирует факт отправки.
- **kafka-offset-reader:** обрабатывает поток событий изменения данных и обновляет пользовательский кэш в Redis.
- **Redis Cluster:** хранит сессионные токены (SessionCache) и предсобранные профили (UserCache), что позволяет всем сервисам быстро проверять авторизацию и получать горячие данные без обращения к основной БД.
- **PostgreSQL:** основное реляционное хранилище для критичных финансовых данных (пользователи, счета, карты, транзакции, платежи, кредиты, вклады).
- **ClickHouse:** колоночная СУБД для хранения логов операций, истории и уведомлений, обеспечивает эффективное сжатие и быструю аналитику.
- **Apache Kafka:** асинхронная шина, которая связывает сервисы: передаёт задания на генерацию выписок, события для уведомлений, поток логов в ClickHouse и команды обновления кэша.
- **S3 (Ceph):** объектное хранилище для готовых квитанций, выписок и временных файлов; клиент получает файл через data‑service по прямой ссылке.

**Потоки данных и балансировка:**  
Внешняя балансировка разделяет трафик: авторизация идёт через Anycast, остальное – через GeoDNS. Внутри ЦОД L4-балансировщик распределяет TCP-соединения на L7 Ingress, который терминирует SSL и направляет запросы по URI в соответствующий микросервис. Авторизация проверяется централизованно – все сервисы обращаются к общему Redis с сессионными токенами. Платёжные операции проходят строгую транзакционную обработку в PostgreSQL и создают события в Kafka. Чтение данных кэшируется в Redis, что снижает нагрузку на базы. Асинхронная генерация выписок реализована через Kafka и отдельный генератор, а клиент оповещается о готовности push‑уведомлением и скачивает файл через data‑service. Остальные ЦОДы имеют идентичную архитектуру, критичные данные реплицируются между площадками.

**Итого на один ЦОД**: 29 серверов (без сетевого оборудования). В двух основных ЦОД: **58 серверов** (это составляет менее 0,1% от общего парка Сбера, который состоит из ~63 тыс. серверов).

**Назначение групп серверов**:  
- **L4‑балансировщик**: принимает весь входящий TCP‑трафик из интернета, равномерно распределяет соединения между L7‑балансировщиками. Пара серверов с общим виртуальным IP обеспечивает мгновенное переключение при отказе.
- **Kubernetes Master**: управляет кластером Kubernetes: хранит состояние в etcd, планирует запуск подов, обрабатывает API‑запросы.
- **Kubernetes Worker**: исполняет контейнеризированные микросервисы (auth, payments, data, products) и системные компоненты. Четыре узла дают запас N+1: даже при выходе из строя одного сервера оставшиеся три вмещают все поды с запасом по ресурсам.
- **PostgreSQL**: хранит критичные транзакционные данные (пользователи, счета, карты, платежи) с соответствием требованиям ACID-транзакций. Шардирование и репликация обеспечивают масштабирование и отказоустойчивость.
- **Redis Cluster**: быстрый кэш в памяти для пользовательских профилей (UserCache) и сессионных данных (SessionCache). Шарды с репликами гарантируют высокую доступность.
- **ClickHouse**: колоночная СУБД для хранения логов операций и уведомлений. Сжатие и партицирование позволяют эффективно хранить колоссальные объёмы данных (актуально для логов).
- **Apache Kafka**: асинхронная шина событий: связывает сервисы, передаёт данные в ClickHouse, триггерит отправку уведомлений и обновление кэша.
- **S3‑хранилище**: объектное хранилище для файлов квитанций, выписок и временных документов. Реплицирует данные внутри ЦОД.
- **Серверы инфраструктуры**: централизованный мониторинг (Prometheus, Grafana), сбор трассировок (Jaeger), логирование (ELK), управление секретами (Vault) и непрерывная доставка (ArgoCD).

**Использование ресурсов worker‑узлов**  
- Каждый worker‑узел: **16 виртуальных CPU, 64 Гб ОЗУ**.  
- Суммарные запросы CPU по всем подам: **38,9 виртуальных CPU** (60% от общей ёмкости четырёх узлов -- 64 виртуальных CPU).  
- При отказе одного узла оставшиеся три предоставляют 48 виртуальных CPU, что с запасом покрывает требуемые 38,9 виртуальных CPU. Таким образом, железо не загружено на 100% ни в штатном режиме, ни при единичном сбое.  
- Запросы памяти: суммарно около **55 Гб**, что также не превышает 86% суммарной ОЗУ четырёх узлов (256 Гб). При отказе узла нагрузка по памяти перераспределяется без превышения ёмкости оставшихся.  
- Поды равномерно распределяются благодаря `podAntiAffinity`, а при росте нагрузки Horizontal Pod Autoscaler (HPA) может увеличить количество реплик в пределах оставшегося запаса ресурсов.
