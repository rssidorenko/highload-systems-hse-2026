# Проект высоконагруженного сервиса Сбербанк Онлайн
## 1. Тема и целевая аудитория
### Описание и аналоги
#### Описание [[1]](https://www.sberbank.ru/ru/person/dist_services)
Сбербанк Онлайн — удаленный канал обслуживания Банка, автоматизированная защищенная система дистанционного обслуживания Клиента через Официальный сайт Банка в сети Интернет, а также Мобильное приложение Банка. Сбербанк Онлайн позволяет получать подробную информацию о банковских услугах: счетах, вкладах, банковских картах и кредитах, а также погашать кредиты, совершать платежи, переводы между вкладами и счетами банковских карт, переводить деньги клиентам Сбербанка и других банков. Помимо этого, в канал обсуживания интегрировано большое количество иных сервисов Сбербанка, включая сервисы для инвестирования, получения документов, контроля и оплаты услуг ЖКХ, использования развлекательных сервисов экосистемы Сбера и пр.
Сервис работает как в виде интернет-кабинета через браузер, так и в виде мобильного приложения для Android и iPhone.

#### Аналоги [[2]](https://www.rustore.ru/prostore/banki-obzor-prilozhenii-1)
У сервиса есть ряд аналогов – как зарубежных, так и отечественных. Основные конкуренты в настоящий момент — отечественные (работа россиян с зарубежными банками осложнена): ТБанк, ВТБ Онлайн, Альфа Банк, Озон Банк, Почта Банк, Точка Банк и др.

### Объём хранилища
+ Среднее количество транзакций в минуту:
  `34 722 + 1 526 + 64 061 = 100 309`
+ Средний размер одного чека равен 60 кБ. Тогда за минуту генерируется:
  `36 248 * 60 кБ = 2 174 880 кБ = 2,074 ГБ` информации, которая заполняет память
+ Объём данных, генерируемых за день транзакциями (сделаем поправку 0,74 на ночное время в часовом поясе МСК) [[12]](https://ru.wikipedia.org/wiki/Население_России):
  `2,074 ГБ * 60 * 24 * 0,74 = 2 210,0544 ГБ`
+ Ежемесячный объём данных по транзакциям:
  `2 210,0544 ГБ * 30 = 66 301,632 ГБ = 64,75 ТБ`
+ Годовой объём данных по транзакциями:
  `64,75 ТБ * 12 = 777 ТБ`
+ Также данные пользователей хранятся в профилях весом 512 кБ:
  `700 кБ * 84 300 000 = 659,48 ТБ`
+ Всего:
  `777 ТБ + 659,49 ТБ = 1436,49 ТБ = 1,4 ПБ`

+ 1.5 - коэффициент увеличения пути (оптоволокно не идет по прямой) [[14]](https://books.google.ru/books?id=5LMp7yxfeDAC&source=ttb&redir_esc=y),
+ 200 - скорость распространения сигнала в оптоволокне (км/мс) [[15]](https://nag.ru/news/18705),
+ 1 мс - статистическая погрешность -> запас на задержки сетевого оборудования.

### Выводы по расположению ЦОД
+ Оптимальное местоположение ЦОД: Центральный и Приволжский ФО, т. к. в них проживает бОльшая доля насления страны (почти 50%).
+ Расположение ЦОД в Центральной России позволяет качественно и без серьёзных задержек обрабатывать даже те данные, которые поступают из наиболее удалённых от них регионов страны (Сибирь, Дальний Восток).
+ Необходимо запланировать строительство новых небольших ЦОД в Сибири и на Дальнем Востоке для возможности экстренного переноса части трафика из ЦФО и ПФО в случае незапланированной остановки их работы.
+ Строительство ЦОД за рубежом благоприятно сказывается на развитии международной сети Сбера

### Схема DNS-балансировки
DNS-балансировка будет использоваться для геораспределенного (GeoDNS) направления пользователей на ближайший региональный кластер или крупный ЦОД.
1. Клиент (мобильное приложение / web-версия) получает IP-адрес устройства пользователя.
2. Запрос поступает в DNS-серверы Сбера, которые определяют примерное местоположение устройства.
3. Клиенту возвращается IP-адрес балансировщика ближайшего ЦОД.
4. В целях обеспечения отказоустойчивости, DNS-записи имеют небольшое время жизни. Это позволит в случае аварии быстро перенаправить трафик, изменив DNS-ответы.

**Схема:** Клиент -> DNS-запрос -> DNS с GeoIP -> Ответ (IP ближайшего ЦОД) -> Клиент -> Запрос к этому IP.

### Схема Anycast-балансировки
Anycast будет применяться для балансировки трафика на уровне критически важных сервисов и точек входа, где требуется максимальная отказоустойчивость и минимизация времени простоя. Например, для серверов авторизации (домен auth).

1. Один и тот же IP-адрес (например, для серверов авторизации) поступает из нескольких ЦОДов одновременно (Москва, Балаково) через протокол BGP.
2. Маршрутизаторы в интернете при помощи протокола BGP направляют запрос клиента к тому ЦОДу, который ближе с точки зрения маршрутизации.

**Схема:**  Пользователь -> Запрос к Anycast IP -> Интернет-маршрутизация (BGP) -> Ближайший доступный ЦОД -> Внутренний балансировщик -> Бэкенд

### Механизм регулировки трафика между ЦОД
1. GSLB (Global Server Load Balancing) на основе DNS
Система GSLB постоянно контролирует состояние и нагрузку на всех балансировщиках в ЦОД.
Алгоритмы:
+ GeoProximity: по умолчанию направляем клиента в ближайший ЦОД;
+ Оценка уровня загрузки: если ближайший ЦОД занят, перенаправляем на следующий
2. Управление с помощью функциональных доменов
+ На уровне балансировки внутри ЦОДа, мы можем регулировать трафик запросов к конкретным микросервисам. Например, прием сложных операций по оформлению кредитов (домен products) можно перенаправлять из ЦОД "Южнопортовый" в более мощный ЦОД "Сколково".

## 4. Локальная балансировка нагрузки
Внутри каждого ЦОД применяется трёхуровневая схема балансировки для обеспечения отказоустойчивости, масштабирования и маршрутизации трафика.

### Уровни балансировки:

1. L4 (Транспортный уровень) – Входной балансировщик (NLB/HAProxy): Первая точка входа для трафика из "внешнего" интернета. Работает на уровне TCP. Его задачи — терминирование SSL и распределение трафика между балансировщиками L7. Использование L4 на входе необходимо для эффективной обработки больших объёмов трафика и защиты внутренних сервисов от TCP-атак.
2. L7 (Прикладной уровень) – Балансировщик HTTP/HTTPS (NGINX/Ingress Controller): Выполняет балансировку на основе HTTP-заголовков, URI и других данных приложения. Здесь происходит маршрутизация запросов к конкретным микросервисам в зависимости от домена (auth, payments, data, products), а также агрегация ответов и ограничение скорости.

Формула резервирования: **N+1**. Такая схема используется, чтобы выдерживать отказ одного узла без потери производительности.

### Расчёт количества балансировщиков
Расчёт выполнен для NGINX как балансировщика L7 [[1]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes)[[2]](https://blog.nginx.org/blog/testing-the-performance-of-nginx-and-nginx-plus-web-servers). Основные ограничители: производительность терминирования SSL (TPS) и пропускная способность сети.

**Метрики производительности балансировщика**

+ Количество запросов SSL/TLS в секунду (TPS): ок. 56 700 (при 16 ядрах).
  Показатель критически важен для домена auth, так как каждая авторизация является новой сессией.

+ Количество запросов HTTPS в секунду (RPS): ок. 340 000 (при 16 ядрах).
  Показатель важен для остальных доменов.

+ Пропускная способность: до 10 Гбит/с.

**Целевые пиковые нагрузки для ЦОД** (на примере ЦФО, где ок. 50% трафика)

Для локальной балансировки и оркестрации контейнеров плнирутся использование сервиса оркестрации Kubernetes. Встроенные механизмы резервирования (ReplicaSets, DaemonSets) гарантируют, что нужное количество балансировщиков всегда будет запущено (формула N+1 реализуется через K8s).

### Выводы
Одного современного балансировщика хватит, чтобы покрыть пиковую нагрузку по всем метрикам. Для обеспечения отказоустойчивости (N+1) и возможности обслуживания мы закладываем 2 балансировщика L7 на каждый крупный ЦОД. Балансировщики L4 (например, на базе Linux Virtual Server или аппаратные) также дублируются в количестве 2 штук на ЦОД. Помимо этого, для большей гарантии работоспособности планируется использование сервиса Kubernetes.
